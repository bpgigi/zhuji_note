1. 项目概览

ZhujiNote 是一款为 MT2026 移动应用测试 课程期末大作业设计的 Android 笔记应用。它以「AI 助记」为核心理念，把一个功能完整的 Markdown 笔记本与 DeepSeek 大模型深度融合，并围绕它系统性地落地了课程 L02–L09 的全部测试能力：开发环境搭建、APK 构建安装、GUI 测试、JUnit 单元测试、MockK Mock 测试、Robolectric + JaCoCo 三阶段覆盖率、Espresso 集成测试、以及 GitHub Actions + Jenkins 双轨持续集成。

与课程前序练习（MobileTestingLab / WeatherLab / IntegrationTestLab）一脉相承，本项目在真实、可运行、功能丰富的工程上完成测试实践，而非为测试而造的玩具应用。全部 225 个测试用例（188 单元 + 37 集成）在本地与 CI 两套环境下 100% 通过。

1.1 核心功能

  • AI 助记（核心）：编辑器内一键调用 9 类 AI 动作——总结 / 续写 / 润色 / 翻译 / 提取关键词 / 起标题 / 生成大纲 / 生成闪卡 / 基于笔记问答，外加独立的多轮 AI 对话页；全部走 DeepSeek SSE 流式，思考过程（reasoning）与最终答案（content）分轨展示。
  • Markdown 编辑器：10 项工具栏（标题 / 加粗 / 斜体 / 代码 / 列表 / 引用 / 链接 / 双向链等）+ 实时预览 + 字数统计。
  • 笔记管理：FTS4 全文检索、标签、置顶收藏、回收站（软删除）、多选批量操作、双向链接 [[笔记]]。
  • 效率工具：番茄钟、写作目标、8 种笔记模板、统计图表（Canvas 自绘）、分享为图片卡片、SAF 备份导入导出、定时提醒。
  • 双主题：Claude 暖橙亮色 + Windsurf 霓虹青暗色，跟随系统或手动切换。

5.5 关键工程问题与解决

问题现象根因与解决
流式卡死AI 只输出几个字就停住OkHttp 默认发 Accept-Encoding: gzip，DeepSeek gzip 压缩 SSE 后 readUtf8Line() 须等整个 block 才出一行。强制 Accept-Encoding: identity 后修复，227 个 chunk 在 4 秒内全部到达。

6. 单元测试三阶段与 JaCoCo 覆盖率（对应 L05–L07）

测试按 stage1 / stage2 / stage3 物理分包，app/build.gradle.kts 通过 afterEvaluate 动态注册三个独立 Test task（按包过滤）与对应 JacocoReport，各阶段写入独立 .exec，再由 jacocoCumulativeReport 合并三份 exec 给出整体覆盖。类目录过滤排除 R / BuildConfig / Hilt_* / *_Impl* / ComposableSingletons / di 生成代码，并整体排除 ui 包（Compose UI 由 Espresso 集成测试覆盖，属业界标准）。

零依赖纯逻辑测试：domain 层 Markdown 解析、TemplateEngine 占位符渲染、BiLinkParser 双向链解析、AiApplyStrategy 语义合并、数据模型往返，全用 assertEquals / assertTrue，无 Mock、无 Android。

7.2 CI 冷启动加固（关键实战）

本项目的 Espresso 用例最初在本地 AVD 全绿，推上 CI 后却有 3 个失败（homeScreenShowsTitle / createNoteAndSeeInList / settingsShowsSaveButton），报 component is not displayed。根因是 CI 环境差异：GitHub 模拟器冷启动慢、视口小，首帧未渲染就断言、或目标按钮落在屏幕外。

解决采用测试最佳实践，抽出 TestExt.kt 通用扩展，并对全部 E2E 用例加固：

fun ComposeTestRule.awaitText(text: String, substring: Boolean = false, timeoutMillis: Long = 20_000) {
    waitUntil(timeoutMillis) {
        onAllNodesWithText(text, substring = substring).fetchSemanticsNodes().isNotEmpty()
    }
}
// @Before 等首页就绪；assertIsDisplayed → assertExists（E2E 只需验证功能可达，不卡视口）
加固手段解决的问题
awaitText / awaitContentDescription冷启动首帧未渲染就断言 → waitUntil 等节点出现
@Before 等首页就绪每个用例起点状态一致
assertExists 替代 assertIsDisplayed视口外的按钮（如「保存并验证」）仍可断言存在
补全 a11y contentDescription首页 FAB、Stats/AiChat 返回按钮原缺无障碍标签，既修测试也修真实缺陷
加固后本地与 CI 双双 37/37 全绿。CI 日志实录：Starting 37 tests → 4/37 → 10/37 → 22/37 → 33/37 completed (0 failed)。这一过程印证了「集成测试必须在目标 CI 环境跑」的价值——本地侥幸通过不等于 CI 通过。
